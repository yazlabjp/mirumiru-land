## 1. 概要とねらい

画面に表示されるポーズの絵(ばんざい等)を見て、児童生徒が全身でそのポーズを真似る。本アプリは**タッチ判定を行わない**唯一のC系アプリで、できた/できないの判定はセンサーではなく**教員が行う**(design.md C-08行「判定は人が行う設計」)。教員は児童生徒の様子を見て、画面下の大きな「できた!」ボタンをタッチして次のポーズへ進める。視覚情報(絵)を全身運動へ変換する力(身体図式)を育てる。

C-01エンジンとの関係: `ctx.addTarget`/`onTouch`によるタップ判定の骨格はC-01と同じだが、タップ対象が「ポーズの絵」ではなく「教員用の確認ボタン1個」である点が異なる(design.md 4-1「C系統の設計メモ」が言う"軽微な拡張"の内容)。

## 2. 設定スキーマ(完全版・このまま実装する)

```javascript
settings: {
  showTime: { type: "choice", label: "まつ じかん", default: "free",
    options: [ {value:"free",label:"せいげんなし"}, {value:"20s",label:"20びょう"},
               {value:"15s",label:"15びょう"}, {value:"10s",label:"10びょう"} ] },
  twoPoses: { type: "toggle", label: "2つ れんぞく", default: false },
  poseCount: { type: "choice", label: "かいすう", default: "n5",
    options: [ {value:"n5",label:"5かい"}, {value:"n8",label:"8かい"},
               {value:"n12",label:"12かい"} ] }
}
```

**設計メモ(規範)**:

- `poseCount`は**予約キー`duration`ではなく通常のchoice設定**として実装する。理由: 本アプリの「ラウンド」は教員の確認待ち(場合により無制限)を含み、`ctx.score`(=できた回数)と表示に出した回数(shown)が一致しない。vt-api.md §13-1の予約解釈(`ctx.remaining = value − ctx.score`)を使うとこの不一致が誤表示になるため、`duration`という名のキーを持たせず、終了判定を完全にアプリ側の`ctx.end()`に委ねる(vt-api.md §13-1冒頭の除外規定どおり)。
- `showTime`は1ポーズあたり教員の確認を待つ**上限時間**。`free`(既定・★1相当)は無制限に待つ。時間切れになっても**児童生徒画面には一切の反応を出さず**、静かに次のポーズへ進む(達成数にはカウントしない。原則3の厳密解釈)。
- `twoPoses`(既定false・★1相当)をtrueにすると、1ラウンドに異なる2つのポーズを同時に表示し、両方できたら1回のボタンタッチでまとめて進める(★2相当)。
- ポーズの絵柄セット自体は本バージョンでは1種類・5ポーズ固定とし、テーマ切り替え設定は持たない(§8参照)。
- digest対象は showTime/twoPoses/poseCount(定義順)。例: `"free/false/n5"`。difficulty:falseの設定はなし。
- 共通設定(sound/reduceMotion/mirror/contrast)はcoreが自動付与する(vt-api §2-2)。**mirrorはこのアプリの見た目に影響しない**。yoko/banzai/kiwotsukeは左右対称のため元々mirrorに意味がない。migi/hidariは向きに意味を持つ唯一の例外だが、この2つは「同じポーズの左右反転」ではなく**別々の名前・キャプションを持つ独立したポーズ**として実装しており(spec-c08.md §3)、mirrorで絵だけを入れ替えるとキャプション(呼び名)と絵が食い違ってしまう。そのためmirrorはこのアプリでは常に無効果とする(C-03/C-04と同様の「無効果」扱い)。「じぶんのみぎ/がめんのみぎ」の視点切り替え(B-05のような動的な左右入れ替え)は本バージョンでは扱わない(§7参照)。reduceMotionはボタン確認時の成功演出のみ簡略化し、ポーズの絵自体は静止画のため影響を受けない。

## 3. ポーズセット(規範・固定5種)

自作SVGで表現し、写真・絵文字は使わない(ai-notes.md §8)。yoko/banzai/kiwotsukeは左右対称。migi/hidariのみ向きに意味を持つが、「同じポーズの左右反転」ではなく**別名・別キャプションを持つ独立した2つのポーズ**として扱う(mirror無効果の根拠。§2設計メモ参照)。

| id | 呼び名 | 姿勢 |
|---|---|---|
| yoko | りょうてよこ | 両腕を肩の高さで真横に伸ばす(Tの字) |
| banzai | ばんざい | 両腕を頭上でVの字に伸ばす(指先まで) |
| kiwotsuke | きをつけ | 両腕を体に沿わせてまっすぐ下ろして立つ |
| migi | みぎでたつ | 画面むかって右の脚で立ち、左の脚を曲げて引き寄せる片足立ち(フラミンゴ立ち)。腕は体幹からわずかに離してバランスを取る |
| hidari | ひだりでたつ | migiの脚を左右反転した片足立ち。画面むかって左の脚で立ち、右の脚を曲げて引き寄せる |

**区別のポイント(実機での指摘を受けた再設計の経緯。2026-07-24)**: 当初のshagamu/hizapon(しゃがむ/ひざぽん)は「やっぱり見分けづらい」との指摘を受け、片足立ち2種(migi/hidari)に差し替えた。kiwotsukeは腕を下げる唯一のポーズ。migi/hidariは脚が左右非対称になる唯一のポーズ(他3種は両脚が対称なまっすぐ立ち)で、この非対称な脚のシルエット自体が残り3種との識別点になる。**ただしmigi/hidariは互いに左右反転の関係にあるため、この2つを並べたときが全ポーズ中もっとも見分けが難しいペアになる**。§5-1で追加したひらがなキャプションが、この2つを区別する主な手がかりとして機能する(絵だけでなく呼び名も参照できる設計)。

## 4. 記録スキーマ

```javascript
record: {
  primary: { key: "completed", label: "できた かいすう", betterIs: "higher" },
  extras:  [ { key: "shown", label: "だした かいすう", unit: "かい" } ]
}
```

- `completed`は「できた!」ボタンが押されて確定したラウンド数(`twoPoses`時も1ラウンド=1カウント)。`shown`は時間切れで進んだラウンドも含む総表示ラウンド数(`shown >= completed`)。
- `shown`はextrasのため先生モード内でのみ表示され、児童生徒画面には出さない(ai-notes.md §0-6)。

## 5. 画面と挙動(共通3画面フローとの差分のみ)

### 5-1. プレイ画面

**実装方式についての前提**: 本アプリは`ctx.addTarget`/`onTouch`を、ポーズの絵ではなく**確認ボタン1個だけ**に対して使う(`stageTouch`は使わない・既定false)。ポーズの絵自体はタッチ対象にしない(触っても無反応)。ポーズ・ボタンの見た目はいずれもVT.Stimに該当がないため、app.js内でSVGを自作する(vt-api.mdに無いVT関数は呼ばない。装飾要素の自作markupのため ai-notes §0-4に抵触しない。spec-c04.md §4-1と同じ扱い)。

**`stageTouch`ではなくボタン単体への`addTarget`を採用した理由**: 検討初期はF-01と同じ全画面ボタン(`stageTouch:true`)の流用も候補にあったが、`stageTouch:true`ではポーズの絵やステージ余白へのタッチも`onTouch(ctx, null)`として拾ってしまい、誤タッチの完全無反応化(原則3・個別DoD8)を保証できない。確認ボタン1個だけを`addTarget`する方式なら、ボタン以外へのタッチはそもそも`onTouch`が呼ばれず無反応が自明に保証されるため、こちらを採用した。

**レイアウト(規範)**:
- 確認ボタン: 直径180px(developer-guide.md §3の主対象120px以上を満たす)の円。`ctx.rect()`の水平中央、下端から40pxのマージンを取った位置(y = `ctx.rect().h - 180 - 40`)に配置。
- ポーズの絵の表示領域: ボタン領域より上、上端24pxマージンを取った範囲に縦方向中央寄せで配置する。`twoPoses:false`のとき1個(幅220×高さ320px相当)を水平中央に、`twoPoses:true`のとき2個(幅170×高さ260px相当・間隔40px)を中央を挟んで左右対称に配置する。
- **ひらがなキャプション**: 各ポーズの絵の下に、§3の「呼び名」をひらがなでキャプション表示する(教員・児童生徒共有の画面でよい)。文字サイズはdeveloper-guide.md §3の児童生徒向け文字の最小規定(24px)を満たす。上記の絵の表示枠(高さ320px/260px)の下端36px分をキャプション表示に充て、絵自体はその残り(284px/224px相当)に収める形で実装してよい(枠の合計高さ・水平中央配置は変えない)。全ラベル(最長「ひだりでたつ」6文字)がポーズ枠の幅(170px/220px)で1行に収まることを確認済み。

**キャプション追加の経緯**: 実機確認で「ポーズの見分けがつきにくい」との指摘を受け、絵のデフォルメ強化に加えて呼び名の表示を追加した(2026-07-24)。教員判定を補助する情報であり、判定そのものを代替するものではない(判定は引き続き教員の目視のみ。§7参照)。

**ラウンド開始(規範)**:
1. `twoPoses:false`なら§3のポーズ5種から1個、`twoPoses:true`なら異なる2個を選ぶ。**直前ラウンドで使ったポーズ(の組)とは異なるもの**を選ぶ(全候補が前回と同じにならない限り再抽選。ai-notes §2の「直前と同じ場所に出さない」規則と同趣旨)。
2. 選んだポーズの絵と確認ボタンをステージに追加し、確認ボタンのみ`ctx.addTarget()`する。
3. `ctx.data.shown += 1`。`showTime`が`free`でなければ、そのms値をカウントダウンする内部タイマーを開始する(表示上のTimerBar等は出さない。§2の設計メモどおり静かに進める)。

**確認ボタンが押されたとき(onTouch)**:
1. `ctx.removeTarget()`でボタンを即座にタッチ対象から外す(連打による二重加算防止。spec-c02.md §4-1実装メモ2と同じ考え方)。
2. `ctx.score += 1`(=completed)。`ctx.feedback.success(buttonEl)`で成功演出+成功音。
3. 演出の再生猶予(750ms。spec-c02.md §4-1のHIT_CLEANUP_MSに準拠)の後、ポーズの絵とボタンをDOMから除去し、`ctx.data.shown`が`poseCount`の数値に達していなければ次ラウンドを開始する。達していれば`ctx.end()`。

**showTime満了時(自動進行)**:
1. 視覚・聴覚とも一切の反応を出さない(原則3)。
2. ボタンを`ctx.removeTarget()`で外し、ポーズの絵とボタンを即座に除去して次ラウンドへ進む(演出待ちが無いため750msの遅延は不要)。`completed`は増やさない。
3. `ctx.data.shown`が`poseCount`の数値に達していれば`ctx.end()`。

**ミスタッチ**: 確認ボタン以外(ポーズの絵・ステージ余白)へのタッチはすべて無視する(`stageTouch`を使わないため、そもそも`onTouch`が呼ばれない)。

**タイマー表示**: `duration`キーを持たないため、coreのTimerBarは表示しない(vt-api §13-1除外規定どおり)。

### 5-2. 先生モードでの表示(差分)

extras(だした かいすう)の直近履歴が表示される。本アプリ固有の追加表示はなし。

### 5-3. 一時停止・設定変更・中断

共通仕様のまま(設定変更ありで閉じたらリスタート、中断は記録なし)。一時停止中はcoreが`onTick`を呼ばないため、showTimeの内部カウントダウンも自動的に停止する。ポーズ・ボタン要素はcoreのシーン破棄で消えるため`onAbort`は省略可。

## 6. 二重符号化・原則3チェック

「できた!」ボタンは常時1種類の見た目・位置のみで、押した/押していないという状態の違いを示す装飾は不要(押した瞬間に成功演出が出て次ラウンドへ進むため、失敗状態という概念自体が画面上に存在しない)。ポーズの絵とボタンはいずれも成功以外の状態を表現しないため、B-01等が対象とする色弁別のような二重符号化の例外には該当しない。なお、migi/hidariは向き(左右)に意味を持つ絵だが、それぞれ独立した呼び名・キャプションを常に伴って表示されるため「色や向きだけで意味の違いを伝える」設計ではない(§2参照)。二重符号化の懸念には当たらない。

## 7. 対象外(本バージョンでやらないこと)

- ポーズセットの複数テーマ切り替え(design.mdの設定候補「ポーズセット」に相当)。v1は§3の5種固定。将来複数セットを追加する場合はdifficulty:falseの`theme`相当設定を追加検討する。
- **B-05との連携(design.mdの発展案「左右指定ポーズ・B-05と連携」)**。§3のmigi/hidari(片足立ち)は見分けやすさのために追加した独立ポーズであり、この発展案が指す「B-05と連動した左右学習モード」とは別物。B-05(みぎひだりタッチ)自体が未実装のため、D-05・E-06のケースと同様に連携仕様は本バージョンでは扱わず、apps.jsonのnext配列にも反映しない。B-05実装後にあらためて連携仕様を検討する。
- 「じぶんのみぎ/がめんのみぎ」といった視点切り替え。migi/hidariは常に「画面むかって右/左」で固定表示し、mirror設定による入れ替えも行わない(§2・§6参照)。
- 体育のサーキットカードとしての大型提示専用モード(design.mdの発展案)。現行UIをそのまま電子黒板・大型TVに映せば代替できるため、専用の表示モードは実装しない。
- カメラ等によるポーズの自動認識。design.md C-08行の「判定は人が行う設計」どおり、教員の目視判定のみを前提とする。

## 8. 受け入れ条件(Definition of Done)

**共通DoD**: ai-notes.md §9 の8項目すべて。

**個別DoD**:
1. ★1相当の設定(showTime=free, twoPoses=false, poseCount=n5)で、教員がボタンを押すまで何秒待っても画面上・音に一切の負のフィードバックが生じない。
2. showTime=10s(または15s/20s)のとき、教員がボタンを押さないまま設定時間が経過すると、負のフィードバックなしに自動的に次のポーズへ進み、completedは増えずshownのみ増える。
3. twoPoses=trueのとき、2つの異なるポーズが同時に表示され、同一ラウンド内で同じポーズが重複しない。
4. poseCount=n5でshownが5に達すると(completedの多寡によらず)結果画面へ遷移し、completed・shownが妥当な値(shown=5、completedはボタンを押した回数)で記録される。
5. 確認ボタンは常に直径180px(120px以上)で表示され、タップすると成功演出(bounce/hanamaru/confetti)と成功音が鳴り、completedが1増える。連打しても二重加算されない(300msデバウンスに加え、押下直後にremoveTargetされるため)。
6. mirror設定をON/OFFしてもポーズの絵の見た目・配置に変化がない(§2設計メモ参照。migi/hidariは向きに意味を持つが、mirrorによる絵の入れ替えは行わない設計)。
7. 結果画面の前回比が、digestまたは日付が異なる直前記録に対して表示されない。
8. iPad実機で、ポーズの絵・確認ボタン以外の場所を連打してもラウンド進行に一切影響しない。
